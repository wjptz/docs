Spec templates give you a head start on the conventions your AI sessions will use on every task. Instead of filling blank spec files from scratch, you start with a curated set of rules for your tech stack — component patterns, API shapes, database conventions, error handling, and common pitfalls — then tailor them to match your actual codebase. Trellis ships three official templates and supports any Git repository as a custom registry, so teams can publish and share their own convention libraries.

<Note>
  Specs are meant to be customized. Every template is a starting point, not a final answer. Always
  replace examples with real file paths from your project and remove rules that do not reflect how
  your team actually works.
</Note>

## How to use a template during init

Pass the `--template` flag to `trellis init` to pull a template directly from the official marketplace:

```bash
trellis init --template electron-fullstack
trellis init --template nextjs-fullstack
trellis init --template cf-workers-fullstack
```

Combine with platform and username flags to set everything up in one command:

```bash
trellis init --template nextjs-fullstack --cursor -u your-name
```

## Custom registries

Available since **v0.3.6**. Use the `--registry` flag to pull templates from any Git repository — your company's internal library, a team fork of the official marketplace, or a community collection.

```bash
# GitHub (public)
trellis init --registry https://github.com/your-org/your-spec-templates

# GitLab
trellis init --registry https://gitlab.com/your-org/your-spec-templates

# Bitbucket
trellis init --registry https://bitbucket.org/your-org/your-spec-templates

# Combine with platform flags
trellis init --registry https://github.com/your-org/your-spec-templates --cursor -u your-name
```

Trellis auto-detects two modes when reading a registry:

- **Marketplace mode** — if the repository contains a `marketplace/index.json` file, Trellis reads the template index and lets you pick which template to install
- **Direct download mode** — if no `index.json` is found, Trellis treats the entire `marketplace/specs/` directory as a single template and downloads it directly

### Handling existing specs

If your project already has specs in `.trellis/spec/`, control how the template merges:

```bash
# Overwrite all existing spec files with template files
trellis init --template nextjs-fullstack --overwrite

# Append template files without touching existing files
trellis init --template nextjs-fullstack --append
```

Use `--append` when you want to add a new stack's conventions alongside ones you have already customized.

## Building a custom spec marketplace

To create a registry others can use with `--registry`, structure your repository like this:

```
your-spec-registry/
└── marketplace/
    ├── index.json
    └── specs/
        ├── my-stack/
        │   ├── frontend/
        │   │   ├── index.md
        │   │   └── components.md
        │   ├── backend/
        │   │   ├── index.md
        │   │   └── api-patterns.md
        │   ├── guides/
        │   │   └── cross-layer-thinking-guide.md
        │   └── README.md
        └── another-stack/
            └── ...
```

The `marketplace/index.json` file lists all available templates:

```json
{
  "version": 1,
  "templates": [
    {
      "id": "my-stack",
      "type": "spec",
      "name": "My Stack Template",
      "description": "Conventions for our internal tech stack",
      "path": "marketplace/specs/my-stack",
      "tags": ["react", "node", "typescript"]
    },
    {
      "id": "another-stack",
      "type": "spec",
      "name": "Another Stack Template",
      "description": "Edge-first API conventions",
      "path": "marketplace/specs/another-stack",
      "tags": ["hono", "cloudflare", "drizzle"]
    }
  ]
}
```

Each `path` points to a directory inside the repo that follows the standard spec structure (`frontend/`, `backend/`, `guides/`, `shared/`, `README.md`). Share the repository URL and teammates install it with `trellis init --registry <url>`.

## Private registries

For private Git repositories, set the `GIGET_AUTH` environment variable to a personal access token before running `trellis init`:

```bash
# GitHub personal access token
GIGET_AUTH=ghp_yourtoken trellis init --registry https://github.com/your-org/private-specs

# GitLab personal access token
GIGET_AUTH=glpat-yourtoken trellis init --registry https://gitlab.com/your-org/private-specs
```

<Tip>
  You don't have to fill every spec file at once. Start with the two or three areas that cause the
  most repeated AI prompting — API patterns, error handling, or component structure — and expand
  over time. The bootstrap task created by `trellis init` guides you through the initial fill.
</Tip>
